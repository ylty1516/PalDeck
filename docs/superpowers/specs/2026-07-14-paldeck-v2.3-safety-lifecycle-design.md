# PalDeck v2.3 安全生命周期设计

## 1. 背景与目标

PalDeck v2.2 已具备本地 PAK、LogicMods、UE4SS Lua Mod、Steam Workshop、固定 Okaetsu UE4SS、匿名 Nexus 目录与响应式桌面界面，但仍有四个生命周期缺口：

1. 本地 Mod 删除成功后会永久清理事务备份，用户无法恢复。
2. 首次扫描接管的外部 Mod 与 PalDeck 安装的 Mod 在界面中没有充分区分。
3. 应用内更新仍信任旧仓库 `ylty1516/palworld-mod-manager`，与正式发布仓库不一致。
4. UE4SS 只能安装或替换，没有所有权审计、安全修复和卸载闭环。

v2.3 的目标是：以统一事务、同卷移动、严格路径派生和可恢复记录补齐上述能力，同时保留 Windows 10/11 Steam 客户端、Nexus 匿名只读、Workshop 不管理订阅生命周期等既有边界。

版本目标为 `2.3.0`。

## 2. 已批准决策

- 使用 **PalDeck 自有回收站**，默认保留 30 天；支持恢复和手动彻底删除。
- 外部发现 Mod 可“取消管理”：保留文件、写入持久忽略清单，后续重扫不再接管。
- 设置页提供“重新发现全部已忽略 Mod”。
- 自动更新可信仓库迁移到 `ylty1516/PalDeck`，不提供兼容任意 URL 的降级路径。
- UE4SS 增加所有权清单、完整性状态、检查并修复、安全卸载；卸载内容进入回收站。
- 删除、恢复、卸载、修复均在游戏停止时执行，并使用现有统一游戏写锁。
- Workshop Mod 继续不提供删除；Steam 仍是订阅与缓存的唯一管理者。

## 3. 方案选择

采用统一安全事务层，而不是 Windows 回收站或仅强化确认。

Windows 回收站无法完整表达 PAK 文件组、禁用文件、UE4SS `mods.txt` 条目和框架所有权，恢复也无法保证事务一致性。统一事务层由三个职责单一的组件构成：

- `TrashService`：回收、恢复、彻底删除和 30 天清理。
- `IgnoredModStore`：外部 Mod 忽略身份和重新发现。
- `Ue4ssFrameworkManager`：UE4SS 所有权、完整性、修复和卸载计划。

现有 `ModService` 继续负责活动 Mod 安装和启停，通过上述组件完成生命周期转换，不在前端拼接文件路径。

## 4. TrashService

### 4.1 存储布局

回收记录保存在：

```text
<data_dir>/trash/records/<trash-id>.json
```

负载必须与源文件位于同一卷，保证 `os.replace` 可用且不会再次触发 `WinError 17`：

- 源文件位于游戏目录时：`<game_root>/.paldeck/trash/<trash-id>/...`
- 源文件位于 PalDeck 数据目录（例如已禁用 Mod）时：`<data_dir>/trash/payload/<trash-id>/...`

一个回收记录可以引用两个同卷负载根，以支持同时拥有游戏文件和 PalDeck 元数据的 UE4SS Mod。记录中的路径不作为任意绝对路径使用；恢复和清理时必须通过位置枚举和当前可信根重新派生。

`.paldeck/trash` 不属于任何 Mod 扫描根，扫描器必须明确忽略 `.paldeck`。

### 4.2 记录模型

每条记录使用 schema version 1，至少包含：

- `id`：规范 UUID。
- `schema_version`：固定为 `1`。
- `entry_type`：`local_mod` 或 `ue4ss_framework`。
- `name`。
- `created_at`、`expires_at`：带时区 ISO-8601；默认相差 30 天。
- `game_fingerprint`：规范化游戏根的摘要，不保存前端提供的路径。
- `files`：原位置枚举、相对路径、负载位置枚举、大小和 SHA-256。
- `manifest`：本地 Mod 的严格清单快照；框架记录为 `null`。
- `ue4ss_state`：需要恢复的 Mod 名称、启用状态或框架内置条目；不保存整份可覆盖的新配置。
- `framework_ownership`：仅框架回收记录保存卸载前的所有权快照。

未知字段、错误类型、越界相对路径、非法 UUID、无时区时间、摘要格式错误或未知位置枚举均 fail-closed。损坏记录可以在 UI 显示为“记录损坏”，但不得执行恢复或清理文件。

### 4.3 移入回收站

本地 Mod 流程：

1. 获取统一游戏写锁并确认 Palworld 未运行。
2. 读取清单并执行现有完整性审计。
3. `modified` 或 `conflict` 状态要求第二次确认；确认后记录实际文件的新大小与摘要。
4. 验证所有源路径和负载根不存在 reparse point。
5. 按源卷将每个文件 `os.replace` 到对应负载目录。
6. UE4SS Mod 仅移除该 Mod 的 `mods.txt` 条目，保留其他行、注释和换行风格。
7. 原子保存回收记录，然后删除活动清单。
8. 任一步骤失败时，反向恢复文件、`mods.txt` 和活动清单，并删除未提交记录。

成功返回 `trash_id`、到期时间和移入文件数。前端文案统一使用“移入回收站”，不再显示“此操作不可撤销”。

### 4.4 恢复

1. 获取游戏写锁并确认游戏未运行。
2. 严格读取记录，重新派生所有源和负载路径。
3. 验证负载文件大小与摘要。
4. 任一恢复目标已存在时，返回 `trash_restore_conflict` 和只读路径列表；不得覆盖。
5. 验证没有活动清单占用同一规范化文件身份。
6. 将文件从同卷负载移回原位置。
7. 恢复本地 Mod 清单；UE4SS 使用当前 `mods.txt` 合并目标条目，不覆盖整份文件。
8. 删除回收记录和空负载目录。
9. 失败时将已恢复文件移回负载，并恢复记录。

### 4.5 彻底删除与过期清理

彻底删除只允许删除由严格记录派生、且位于两个受控负载根中的文件。先删除负载，再删除记录；负载清理失败时保留记录并标记错误，便于重试。

应用启动后和打开回收站时执行过期清理。损坏记录、正在处理的记录或负载校验失败的记录不自动删除。清理不得要求游戏运行状态，因为负载不在游戏加载路径内，但仍使用 TrashService 自身的跨进程锁避免并发恢复。

## 5. 外部发现与忽略清单

### 5.1 来源标识

现有 `source_name == "rescan"` 的清单视为外部发现；PalDeck 导入的清单继续视为 PalDeck 安装。API 增加派生布尔字段 `externally_discovered`，前端不得自行猜测路径。

界面显示：

- `外部发现`：黄色来源标签，说明“安装 PalDeck 前已存在”。
- `PalDeck 安装`：绿色来源标签。

两类 Mod 都使用“移入回收站”。只有外部发现 Mod 显示“取消管理”。

### 5.2 IgnoredModStore

存储文件：

```text
<data_dir>/ignored-mods-v1.json
```

按 `game_fingerprint` 分组。身份由以下后端可信字段构成：

- `kind`。
- 安装根枚举：`tilde_mods`、`logic_mods`、`ue4ss_classic`、`ue4ss_nested`。
- PAK/LogicMods 使用规范化文件组名称；UE4SS 使用规范化目录名称。

不存储或接受前端提供的任意本地路径。

### 5.3 取消管理

仅允许 `externally_discovered == true` 的活动清单：

1. 获取游戏锁并确认游戏停止。
2. 验证文件审计不是 `missing` 或 `conflict`；异常时返回明确错误，不删除清单。
3. 先写入忽略身份。
4. PAK/LogicMods 删除清单但不移动文件。
5. UE4SS 保留当前 `mods.txt` 条目；若 PalDeck 接管时保存过 `enabled.txt` 元数据，则通过复制、摘要校验和原子发布恢复到 Mod 目录，再删除 PalDeck 元数据。
6. 删除清单。失败时删除忽略身份并恢复元数据。

扫描 `rescan()` 在创建清单前检查忽略身份。设置页“重新发现全部”只清空当前游戏的忽略项，然后立即执行安全重扫；游戏运行时返回 423，不提前清空。

## 6. 自动更新仓库迁移

以下默认值统一改为：

```text
owner: ylty1516
repo: PalDeck
```

涉及 `backend/self_updater.py`、`backend/version.py`、测试、README 和发布契约。

安全要求不变并加强：

- 只查询 `https://api.github.com/repos/ylty1516/PalDeck/releases/latest`。
- 只接受 `github.com/ylty1516/PalDeck/releases/download/...` 及经过现有白名单验证的 GitHub 资产重定向。
- 资产必须为 PalDeck EXE 或便携 ZIP。
- 必须存在同名 `.sha256` sidecar，内容只能是一条严格摘要与资产名。
- 下载后使用常量时间摘要比较。
- 缺失 sidecar、仓库不符、相似域名、资产名不符、摘要错误均拒绝更新。
- 更新仅替换当前 EXE，保留 `data`、回收记录、忽略清单、背景和配置。

旧仓库不作为网络降级源。v2.2.4 用户需要手动升级到 v2.3.0；从 v2.3.0 起应用内更新跟随 PalDeck 正式仓库。

## 7. Ue4ssFrameworkManager

### 7.1 所有权存储

存储文件：

```text
<data_dir>/ue4ss-framework-v1.json
```

按 `game_fingerprint` 保存：

- schema version。
- 安装来源：`bundled`、`upstream` 或 `local_zip`。
- 资产名称、固定摘要和安装时间。
- 每个受管文件相对 `Win64` 的规范化路径、大小和安装后 SHA-256。
- `mutable` 标志。
- 包内框架 Mod 名称集合。

安装器返回服务端生成的文件记录；前端不能提交所有权路径。所有路径必须通过 Win64 根、Windows 保留名、ADS、路径穿越和 reparse 校验。

### 7.2 可变文件

以下文件视为可变状态：

- classic/nested `UE4SS-settings.ini`。
- classic/nested `Mods/mods.txt`。
- classic/nested `Mods/mods.json`。

首次安装缺失时从包创建。修复已有安装时：

- settings 保留全部用户配置，只确保 Palworld 必需的 `bUseUObjectArrayCache = false`。
- `mods.txt` 保留用户行、注释、顺序和换行；只补充包内框架缺失条目，已有值不覆盖。
- `mods.json` 已存在时不覆盖，缺失时从包恢复。

可变文件参与状态说明，但安全卸载不删除它们。

### 7.3 状态和完整性

`GET /api/ue4ss/status` 在现有字段基础上增加：

- `managed`：是否存在有效所有权清单。
- `integrity`：`not_installed`、`unmanaged`、`healthy`、`missing`、`modified` 或 `conflict`。
- `owned_files`、`missing_files`、`modified_files` 数量。
- `repair_available`、`uninstall_available`。

旧版本已安装但没有所有权清单时显示 `unmanaged`。此状态只允许“检查并修复”建立清单，或按固定内置包已知路径执行二次确认卸载；未知文件永不纳入计划。

### 7.4 检查并修复

修复固定使用已审计内置 Okaetsu 包：

1. 获取游戏锁，拒绝活动 Workshop UE4SS。
2. 验证内置资产摘要与布局。
3. 根据所有权清单或固定包路径检查核心文件。
4. 列出缺失、修改和冲突；修改文件需要确认后覆盖。
5. 在游戏卷创建同卷事务目录，备份将替换的核心文件。
6. 发布核心文件，并按 7.2 合并可变文件。
7. 原子保存新的所有权清单。
8. 失败时恢复文件、可变配置和旧所有权清单。

修复不修改用户 UE4SS Mod 目录中不属于固定包的文件。

### 7.5 安全卸载

1. 获取游戏锁并确认游戏停止。
2. 从有效所有权清单构建精确计划；无清单时只使用固定内置包的已知核心路径。
3. 对修改过的受管核心文件要求 `confirm_modified`。
4. 仅将非可变的精确受管文件移入 TrashService。
5. 从 `mods.txt` 删除固定包框架 Mod 的条目，保留用户条目、注释和换行。
6. 不删除 settings、mods.json、用户 Mod 文件或未知文件；仅清理空目录。
7. 将所有权快照和框架条目状态保存到 `ue4ss_framework` 回收记录。
8. 删除活动所有权记录。失败时完整回滚。

恢复框架回收记录时，目标冲突规则与本地 Mod 相同，并将缺失的框架条目合并回当前 `mods.txt`。

## 8. API 设计

所有新接口沿用现有 Cookie 会话、标准 `{ok, data}` 包络和固定字段验证。

### 回收站

- `GET /api/trash`：列出当前游戏的有效与损坏记录，不返回负载绝对路径。
- `POST /api/trash/<trash_id>/restore`，body 必须为 `{}`。
- `DELETE /api/trash/<trash_id>`：彻底删除。
- 现有 `DELETE /api/mods/<mod_id>` 改为移入回收站，保留 `force_modified=true`，返回回收记录摘要。

### 外部 Mod

- `POST /api/mods/<mod_id>/unmanage`，body 必须为 `{}`。
- `GET /api/mods/ignored`：返回当前游戏忽略数量和安全身份摘要。
- `POST /api/mods/ignored/reset`，body 必须为 `{}`；清空后重扫。

### UE4SS

- `POST /api/ue4ss/repair`：仅接受 `confirm_replace` 布尔值。
- `POST /api/ue4ss/uninstall`：仅接受 `confirm_modified` 布尔值。
- `GET /api/ue4ss/status` 扩展第 7.3 节字段。
- 现有内置、本地 ZIP 和上游安装成功后均更新所有权清单。

错误码至少包括：

- `trash_record_invalid`
- `trash_restore_conflict`
- `trash_payload_missing`
- `trash_payload_modified`
- `mod_not_external`
- `ignored_reset_requires_game_stopped`
- `ue4ss_repair_conflict`
- `ue4ss_modified_files`
- 现有 `game_running`、`permission_denied`、`ue4ss_conflict`

## 9. 前端交互

### 我的模组

- 来源单元显示 `外部发现` 或 `PalDeck 安装`。
- 删除按钮文案改为“移入回收站”。
- 外部发现 Mod 增加“取消管理”。
- 顶部增加“回收站 N”入口，打开响应式对话框或抽屉。

### 回收站

每项显示名称、类型、文件数量、删除时间和剩余天数。操作为“恢复”和“彻底删除”。彻底删除必须二次确认，明确不可恢复。记录损坏时只显示诊断状态和安全的“删除记录”建议，不执行负载路径操作。

### 设置与 UE4SS

UE4SS 卡片显示：安装状态、所有权状态、受管文件数、缺失/修改数。按钮按状态展示：

- 未安装：安装内置、检查 GitHub、本地 ZIP。
- 未受管/异常：检查并修复、安全卸载。
- 健康：检查并修复、安全卸载、检查 GitHub、本地 ZIP。

卸载说明明确“用户 Mod 和配置会保留，框架进入回收站”。

高级设置增加“重新发现已忽略 Mod”，同时显示当前忽略数量。

所有长操作显示阶段：检查 → 备份/回收 → 发布/恢复 → 更新清单。错误同时写入当前结果区域和 toast。

## 10. 迁移与兼容

- 现有 Mod 清单不批量重写，通过 `source_name == "rescan"` 派生外部来源。
- 现有数据目录结构保留；新 JSON 文件按需创建。
- 损坏新存储 fail-closed，不影响只读列出现有 Mod，但禁止危险写操作。
- 已有 UE4SS 无所有权清单时显示 `unmanaged`，不自动宣称文件属于 PalDeck。
- Workshop、主题、背景、Nexus 缓存、视觉基线和应用会话保持兼容。
- 回收站负载不打包进更新 ZIP，不随应用更新删除。

## 11. 测试策略

### TrashService

- 启用/禁用 PAK、LogicMods、UE4SS Mod 的回收和恢复。
- 游戏盘与数据盘不同的跨盘组合。
- 恢复目标冲突、负载缺失、负载修改、记录损坏。
- 修改文件二次确认。
- 事务各发布点故障注入和完整回滚。
- 30 天到期、手动彻底删除、清理失败重试。
- reparse、ADS、路径穿越和伪造位置枚举拒绝。

### 外部发现

- 来源派生、取消管理、忽略后重扫、清空后重新发现。
- PAK 文件组和 classic/nested UE4SS。
- `enabled.txt` 恢复与 `mods.txt` 状态保持。
- 非外部 Mod 拒绝取消管理。

### 更新器

- API 与下载 URL 仅接受 `ylty1516/PalDeck`。
- 旧仓库、相似域名、任意 URL、非法重定向拒绝。
- ZIP/EXE 选择、sidecar 缺失、格式错误、摘要错误。
- 更新后 `data` 保持。

### UE4SS

- 三种安装来源生成所有权清单。
- healthy/missing/modified/unmanaged 状态。
- 修复保留 settings、mods.txt 用户行和 mods.json。
- 卸载只回收受管非可变文件。
- 无清单固定路径识别、修改确认、用户 Mod 保留。
- 跨盘、失败回滚、回收后恢复。
- Workshop UE4SS 互斥和统一游戏锁并发测试。

### 前端与发布

- 动态 action、ARIA、焦点恢复、确认语义和阶段反馈契约。
- 五页 × `1600×1000`、`1280×820`、`1236×771`、`960×640` 共 20 张视觉基线。
- Frozen smoke 增加回收站、外部标签、新更新仓库、UE4SS 修复/卸载标记。
- 完整 Python、Node、语法、compileall、安全边界、两次可复现构建、ZIP CRC、sidecar 和 GitHub 回下载验证。

## 12. 发布与成功标准

发布 `v2.3.0` 前必须满足：

1. 删除的本地 Mod 可在 30 天内完整恢复，且跨盘不报错。
2. 外部发现 Mod 可持久取消管理，并可由用户主动重新发现。
3. 应用内更新只信任 `ylty1516/PalDeck`。
4. UE4SS 可审计、修复、卸载和从回收站恢复，未知用户文件不被删除。
5. 所有危险操作在游戏运行、路径异常、记录损坏或恢复冲突时 fail-closed。
6. 自动化、20 张视觉基线、Frozen smoke、两次构建和发布后回下载全部通过。
