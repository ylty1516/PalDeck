# PalDeck v2.2：概念图高保真界面重构设计

## 1. 目标

PalDeck v2.2 以用户提供的五张 `1600×1000` 概念图为视觉基准，重构“我的模组、导入安装、N 网热门、设置与外观、开源致谢”五个页面，在不削弱真实管理能力和安全边界的前提下实现统一的蓝白磨砂玻璃桌面界面。

成功标准：

1. 在 `1600×1000` 窗口中，整体空间关系、视觉层级、圆角、磨砂、按钮、列表和底栏与概念图高度一致。
2. 保持当前 `1280×820` 默认窗口和 `960×640` 最小窗口；三档尺寸均无横向滚动、遮挡或不可达操作。
3. 保留本地 Mod、Workshop、UE4SS、事务安装、冲突处理、应用更新、主题、背景、樱花和开源致谢的真实功能。
4. 所有数字、状态、按钮和链接均由真实数据驱动；不复制概念图中的虚构统计、项目、贡献者或操作。
5. Nexus 继续匿名只读并彻底过滤成人内容，不增加登录、收藏、下载或一键安装。
6. 版本升级为 `2.2.0`，通过自动化契约、三档布局截图、Frozen smoke 与可复现构建后发布正式便携版。

## 2. 非目标

- 不改写 Python + Flask + pywebview 架构，不引入 Electron、Vue、React 或前端构建链。
- 不改变 Mod、Workshop、UE4SS 的磁盘语义、事务协议、互斥规则和共享游戏锁。
- 不增加 Nexus 登录、收藏、endorse、下载或自动安装。
- 不展示成人 Nexus Mod，也不提供关闭过滤的设置。
- 不使用概念图里虚构的 Mod、Nexus 数据、贡献者、依赖、版本或日期。
- 不把五张完整概念图或其中的角色图像裁切后作为应用资产；它们仅作为设计参考。
- 不删除设置页中概念图未展示的现有功能。
- 不迁移应用内自更新的受信任仓库；仍保持当前固定仓库和校验策略。
- 不支持 Xbox/Microsoft Store 或专用服务器管理。

## 3. 已确认决策

- 采用“忠实响应式重构”，而不是固定 `1600×1000` 画布或仅替换 CSS 皮肤。
- 默认窗口继续使用 `1280×820`，不启动最大化。
- Workshop、UE4SS、冲突修复、更新和完整外观设置全部保留；次要内容进入可访问的折叠分组。
- Nexus 概念图中的“收藏/一键安装”替换为真实可用的“打开 N 网/复制尾号”。
- 应用标识继续使用 PalDeck 自有 `assets/app.png`，不复制概念图中的 Pal 角色 Logo。
- 本地线性 SVG 图标随应用分发；不从 CDN、远程字体或在线图标服务加载资源。
- 宽屏使用横向底部状态栏；紧凑尺寸压缩信息，但不在侧栏和底栏重复同一个交互控件。
- 发布 ZIP 复制到 `C:\Users\yyx\Desktop\pal mod`，不再放到桌面根目录。

## 4. 视觉系统

### 4.1 总体风格

视觉方向为清透蓝白玻璃，不改变现有三套主题的可选能力：

- 默认“蓝白玻璃”主题对应概念图：高亮蓝 `#1677ff`、深海军蓝正文、半透明白面板、淡蓝描边和柔和蓝灰阴影。
- 现有深色和象牙主题继续可选，沿用相同组件几何和布局，只替换颜色令牌。
- 背景覆盖整个客户区；内置背景和用户上传背景继续由现有 Appearance API 提供。
- 玻璃效果由半透明底色、细描边、阴影和受限 `backdrop-filter` 组合；低性能或不支持模糊时退化为高不透明度面板，不影响可读性。

统一几何令牌：

- 外壳间距：宽屏 `24px`，默认窗口 `16px`，紧凑窗口 `12px`。
- 大面板圆角：`22–26px`；普通卡片：`14–18px`；输入框和按钮：`10–12px`。
- 主操作使用纯蓝渐变或纯蓝底，次操作使用白色玻璃底和蓝灰描边，危险操作保持红色语义。
- 标题字重 `700–800`，正文 `400–500`；使用 Windows 系统中文字体栈，不远程加载字体。
- 图标统一为 `20/24px` 线性 SVG，状态图标保留蓝、绿、橙、红四类语义色。

### 4.2 三段式应用外壳

客户区使用 CSS Grid：

1. 左侧导航栏：Logo、五项导航、自定义背景入口和版本/更新入口。
2. 右侧主内容：页面标题、页面级操作和可滚动内容。
3. 底部状态栏：服务状态、游戏目录和配置/诊断入口。

宽度规则：

- `>=1440px`：侧栏约 `300px`，底栏约 `68px`，内容按概念图充分展开。
- `1100–1439px`：侧栏约 `236px`，底栏约 `62px`，降低卡片列数和水平间距。
- `960–1099px`：侧栏约 `204px`，底栏保持单行紧凑状态；双栏内容改为单栏或主次栏。
- 浏览器调试宽度低于 `960px` 时允许导航折叠，但 Frozen Windows 应用仍以 `960×640` 为支持下限。

主内容自身滚动，侧栏和底栏固定；页面切换不改变外壳尺寸。樱花 Canvas 位于背景上、玻璃层下，不拦截鼠标。

### 4.3 窗口外观

Windows 构建采用与概念图一致的无边框客户区，在右上角提供最小化、最大化/还原和关闭按钮，并提供明确拖拽区域：

- 窗口控制只调用本地 pywebview 窗口对象，不接收任意命令或路径。
- 拖拽区域不覆盖输入框、按钮、滚动区和导航。
- 最大化后保留 Windows 工作区边界，不覆盖任务栏。
- `Esc` 不关闭主窗口；关闭仍走现有退出与更新替换流程。
- 若 WebView2/pywebview 在目标系统无法启用无边框拖拽，启动时回退到原生 Windows 标题栏，功能优先于装饰。

## 5. 五个页面

### 5.1 我的模组

页面结构：标题与刷新/打开模组目录操作、搜索与筛选栏、真实统计卡、统一列表。

- 搜索匹配名称、类型和 Nexus ID；筛选覆盖来源、启用状态和异常状态。
- 统计只显示可从当前列表计算的总数、已启用、已禁用和异常/冲突数；不存在真实数据时不显示“可更新”。
- 本地 Mod 与 Workshop Mod 使用同一视觉行，但来源徽标和操作集不同。
- 本地 Mod 保留启停、打开目录、重扫和删除；Workshop 保留启停、打开 Workshop 目录和 Steam 页面。
- 完整性异常、缺失依赖、环、待重启、部署状态和清理待完成均使用现有真实字段，不用颜色作为唯一提示。
- 行高允许名称和路径省略；悬停或聚焦时提供完整文本，操作按钮始终可见。

### 5.2 导入安装

页面结构：拖放区、快捷选择、路径检测、安装预检/结果、真实待处理项。

- 支持现有 ZIP、PAK 和文件夹入口；文件夹入口只枚举用户通过原生选择器选定目录顶层的普通 `.zip`/`.pak` 文件，按稳定文件名顺序加入队列，不递归、不跟随 reparse point 或符号链接，也不直接安装目录。文件类型、大小和目标来自真实预检。
- 不伪造概念图中的四项导入队列。选择多个受支持输入时，前端按顺序预检并逐项要求确认；每个文件仍独立复用现有冲突处理和事务安装语义，单个输入只显示一个真实待处理项。
- 安装过程明确显示“选择、预检、冲突确认、事务发布”四阶段，但阶段状态必须来自当前操作。
- 右侧详情展示当前选中项的来源、类型、目标、文件清单、依赖和冲突；缺失字段显示“未提供”，不推断作者或描述。
- 冲突继续使用既有确认弹窗和后端事务语义；失败时保留错误详情和重试入口，不把失败项标记为成功。

### 5.3 N 网热门

页面结构：搜索、排序标签、来源状态、响应式目录卡片和辅助侧栏。

- 排序只提供当前 API 支持的热门下载、热门推荐和最新发布。
- 搜索同时支持关键词和纯数字 Nexus ID；纯数字走固定游戏的 Mod ID 查询。
- 卡片展示真实图片、名称、作者、版本、下载、推荐和 Nexus ID；缺失图片使用本地中性占位。
- 操作固定为“打开 N 网”和“复制尾号”。不显示收藏、评分写入、一键安装或下载进度。
- 右侧辅助区提供 ID 直达、数据来源/缓存状态、匿名只读说明和成人内容过滤说明；不显示虚构“编辑精选”。
- 后端继续在 live、fresh cache、stale cache 和单 ID 查询上 fail-closed 过滤成人内容；前端继续在读取敏感字段前做防御性跳过。

### 5.4 设置与外观

页面使用概念图的卡片网格和实时效果预览，并将现有能力分为：

1. 游戏目录：自动检测、手动路径、打开目录和修复目录。
2. 启动与更新：应用更新检查和现有启动相关选项。
3. UE4SS：安装状态、内置安装、本地 ZIP、固定上游检查和互斥提示。
4. 外观主题：三套主题缩略预览。
5. 背景自定义：上传、恢复默认、遮罩、模糊、位置和缩放。
6. 动效：樱花密度、自然/水彩/极简风格、reduced-motion 状态。
7. 高级选项：不适合主卡片的诊断和次要控制。

预览使用轻量 DOM 示意，不嵌套真实应用或重复执行 API；主题和背景更改继续遵循 revision/串行写入保护。需要重启才生效的设置必须明确标注。

## 6. 组件与代码边界

继续使用原生 ES Modules，按职责拆分而不引入框架：

- `index.html`：稳定语义骨架、五视图容器、窗口控件、底栏和弹窗。
- `styles.css`：主题令牌、应用外壳、通用组件和页面布局；若文件过大，可按 `shell/components/views/themes` 拆成静态 CSS 文件后由 HTML 直接加载。
- `render.js`：安全创建本地 Mod、Workshop 和 Nexus 的动态列表/卡片，不使用 `innerHTML` 注入远端内容。
- `app.js`：状态编排、视图切换、筛选、真实统计、加载/错误状态和现有动作委托。
- 新增 `icons.js` 或静态 SVG sprite：只导出固定本地图标，不接受远端 SVG/HTML。
- 新增小型 `desktop_bridge` pywebview bridge：仅提供最小化、最大化/还原、关闭、窗口状态查询，以及由原生目录选择器产生的顶层 ZIP/PAK 清单；不接受前端提交任意路径。

现有 API 路径、Token 校验、可信 URL、文件事务和 Mod 数据结构保持兼容。视觉层需要派生字段时在前端纯函数中计算，不修改后端业务对象以迎合布局。

## 7. 状态与数据流

1. 应用启动后加载 Appearance、服务状态、游戏路径和当前视图所需数据。
2. 外壳立即显示；内容区使用与最终布局等高的骨架屏，避免数据返回后大幅跳动。
3. 页面 API 成功后以安全 DOM API 渲染真实数据并计算统计。
4. 用户筛选只作用于已加载列表；刷新与变更操作成功后重新读取后端真相，不乐观伪造最终磁盘状态。
5. 页面切换保留各页搜索和折叠状态，但不保留过期的安装确认或冲突授权。
6. 背景、主题和樱花沿用现有 revision 与队列控制，防止迟到响应覆盖新选择。

## 8. 错误、空状态与可访问性

- 每个页面定义加载、空、成功、部分数据、错误和 stale cache 状态。
- 网络错误、路径错误、锁冲突、游戏运行中、UE4SS 互斥和事务回滚保留具体可执行提示。
- Toast 只用于短期结果；需要决定或修复的问题放在内容区或模态框，不能自动消失。
- 所有图标按钮具有中文可访问名称、可见焦点和 tooltip；完整操作支持键盘。
- 导航同步 `aria-current="page"`；折叠区同步 `aria-expanded`；模态框管理焦点进入和返回。
- 颜色对比满足普通文本至少 4.5:1；状态同时使用文字/图标，不只靠颜色。
- `prefers-reduced-motion` 下关闭樱花和大幅位移动画，仅保留必要的短淡入。
- 远程 Nexus 图片保持 `https`、懒加载、无 referrer、失败移除和不执行内容。

## 9. 性能设计

- 不加载远程字体、CDN CSS、远程图标或背景脚本。
- SVG 图标去重；缩略图按显示尺寸加载，Nexus 图片继续懒加载。
- `backdrop-filter` 只用于外壳级玻璃面板，不对每个列表行叠加大半径模糊。
- 长列表使用稳定 DOM 更新；初版不引入虚拟滚动，只有真实性能测量显示需要时才增加。
- 樱花继续沿用粒子数、DPR、hidden/reduced-motion、resize 和销毁上限。
- 在目标 Windows WebView2 上验证默认窗口空闲和滚动时无明显卡顿。

## 12. 风险与缓解

- **概念图是生成式静态稿：** 其中含虚构数据和不符合当前架构的 Electron/Vue 内容。只复刻视觉语言，数据和依赖以产品真相为准。
- **无边框窗口兼容性：** 在 Windows 10/11 与不同 DPI 上验证拖拽和控制按钮；失败时自动使用原生标题栏。
- **玻璃效果性能：** 控制模糊层数量并提供无模糊降级，不将滤镜应用到长列表每一行。
- **默认窗口内容密度：** 使用明确断点与折叠高级内容，不通过缩小字号强塞概念图全部内容。
- **真实数据长度不可控：** 名称、路径和作者采用省略、换行和可访问完整提示，测试极长中文、英文和路径。
- **视觉改动影响动作契约：** 保持稳定 ID/data-action 或同步迁移测试，逐页验证真实文件操作而非仅检查静态页面。
